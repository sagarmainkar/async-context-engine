# Async Context Engine Library Design

**Date:** 2026-03-01
**Status:** Approved

## Goal

Turn the async context engine into a pip-installable, open-source Python library that gives developers plug-and-play async context re-entry for LangGraph agents. Developers import the library, extend the state, dispatch tasks, and get automatic result delivery — without understanding the re-entry mechanics.

## Key Decisions

| Decision | Choice |
|----------|--------|
| Task execution | Developers manage; library tracks + re-enters |
| Classification | Explicit `dispatch_task()` — no auto-classify |
| Task record | thread_id, task_id, status, result, created_at, updated_at, error |
| Backend API | ABC with CRUD methods (`TaskStore`) |
| Checkpointer | Separate from LangGraph checkpointer |
| Poller | Built into library, not optional |
| Distribution | Pip-installable package |

## Core Data Model

### TaskRecord

```python
@dataclass
class TaskRecord:
    task_id: str           # UUID, generated by library
    thread_id: str         # Links task to a conversation/session
    status: str            # "pending" | "running" | "completed" | "failed"
    description: str       # Human-readable description
    result: str | None     # Result data when completed
    error: str | None      # Error info when failed
    created_at: datetime
    updated_at: datetime
```

### AsyncTaskState (mixin)

```python
class AsyncTaskState(TypedDict):
    task_records: dict[str, TaskRecord]   # task_id -> TaskRecord
    results_buffer: list[dict]             # Completed results awaiting delivery
```

Developers compose into their own state:

```python
class MyState(AsyncTaskState):
    messages: Annotated[list[dict], operator.add]
    # ... their own fields
```

## TaskStore ABC (Persistence Interface)

```python
class TaskStore(ABC):
    @abstractmethod
    def create_task(self, record: TaskRecord) -> None: ...

    @abstractmethod
    def update_task(self, task_id: str, status: str,
                    result: str | None = None,
                    error: str | None = None) -> None: ...

    @abstractmethod
    def get_task(self, task_id: str) -> TaskRecord | None: ...

    @abstractmethod
    def get_tasks_by_thread(self, thread_id: str) -> list[TaskRecord]: ...

    @abstractmethod
    def get_tasks_by_status(self, thread_id: str, status: str) -> list[TaskRecord]: ...
```

**Shipped implementations:**
- `InMemoryTaskStore` — dict-based, for unit tests and quick demos
- `FileTaskStore` — JSON file-based, for proof-of-concept demos

Developers implement `TaskStore` for Postgres, Redis, DynamoDB, etc. as needed.

## Dispatch & Re-entry Flow

### Step 1: Dispatch from graph node

```python
from async_context_engine import dispatch_task

def my_dispatcher_node(state, config):
    thread_id = config["configurable"]["thread_id"]

    # Library generates task_id, persists as "pending"
    task = dispatch_task(store, thread_id, description="Process order Alpha")

    # Developer sends task_id + thread_id to their external system
    my_kafka_producer.send(topic="orders", value={
        "task_id": task.task_id,
        "thread_id": thread_id,
        "payload": {"order": "Alpha"},
    })

    return {
        "messages": [{"role": "assistant", "content": f"On it! ({task.task_id})"}],
        "task_records": {task.task_id: task},
    }
```

### Step 2: External system reports result (standalone helper)

```python
# In a completely different process/service
from async_context_engine import update_task_result
from async_context_engine.stores import FileTaskStore

store = FileTaskStore(path="/shared/tasks.json")

def on_order_processed(event):
    update_task_result(
        store=store,
        task_id=event["task_id"],
        result=f"Order total: {event['total']}",
    )
    # No graph, no config, no LangGraph knowledge needed.
```

### Step 3: Built-in poller handles re-entry

The poller is a core part of the library. It:
- Runs as a daemon thread at a configurable cadence
- Queries the TaskStore for the thread_id's tasks
- Finds newly completed tasks
- Patches results into the graph's `results_buffer` via `graph.update_state()`
- Triggers a graph run with a synthetic system message
- The conductor picks up results and delivers them to the user

```python
from async_context_engine import AsyncPoller

poller = AsyncPoller(store=store, graph=compiled_graph, config=config, interval=5)
poller.start()
```

### Step 4: Classifier checks for results

```python
from async_context_engine import has_pending_results

def my_classifier(state) -> str:
    if has_pending_results(state):
        return "conductor"
    # ... normal routing
```

## Package Structure

```
async_context_engine/
├── __init__.py          # Public API exports
├── models.py            # TaskRecord dataclass
├── state.py             # AsyncTaskState TypedDict mixin
├── dispatch.py          # dispatch_task(), update_task_result()
├── poller.py            # AsyncPoller (daemon thread)
├── store/
│   ├── __init__.py      # TaskStore ABC
│   ├── memory.py        # InMemoryTaskStore
│   └── file.py          # FileTaskStore (JSON-based)
└── helpers.py           # has_pending_results() and utilities
```

## Developer Integration (4 steps)

```python
# 1. Define state (extend the mixin)
class MyState(AsyncTaskState):
    messages: Annotated[list[dict], operator.add]

# 2. Configure store
store = FileTaskStore(path="./tasks.json")

# 3. In your graph node, dispatch tasks
task = dispatch_task(store, thread_id, description="...")

# 4. Start the poller
poller = AsyncPoller(store=store, graph=graph, config=config, interval=5)
poller.start()
```

## Examples

### examples/basic/

Current project refactored to use the library. Fully working demo with simulated background tasks and FileTaskStore. Rich CLI with interactive conversation, async dispatch, and proactive result delivery.

### examples/sub-agents/

Skeleton showing sub-agent integration. The main graph dispatches a task and fires off a sub-agent in a different system to do the work. The sub-agent may involve 2-way communication (e.g., querying external APIs, coordinating with other services). When the sub-agent completes, it calls `update_task_result()` to report back. Comments indicate where real sub-agent orchestration code goes.

### examples/multi-agent/

Skeleton showing a multi-agent setup where the main agent dispatches tasks to multiple specialized sub-agents running in different processes or systems. Each sub-agent works independently and reports results back via `update_task_result()`. Demonstrates fan-out/fan-in patterns with the library.

## Architecture Diagram

```
┌─────────────────────────────────────────────────────┐
│                Developer's Graph                     │
│                                                      │
│  [User Input] → [Classifier] → [Conductor] → [Out]  │
│                      │              ↑                │
│                      │              │                │
│                 [Dispatcher]   results_buffer         │
│                      │              ↑                │
└──────────────────────┼──────────────┼────────────────┘
                       │              │
              dispatch_task()    AsyncPoller
                       │         (built-in)
                       ↓              ↑
              ┌────────────────────────────┐
              │        TaskStore           │
              │  (File / Memory / Custom)  │
              └────────────────────────────┘
                       ↑
              update_task_result()
                       │
              ┌──────────────────────────┐
              │  External Sub-Agents     │
              │  (other systems, APIs,   │
              │   2-way communication)   │
              └──────────────────────────┘
```
